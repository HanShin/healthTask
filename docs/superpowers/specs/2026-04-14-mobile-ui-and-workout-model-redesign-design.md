# 오늘운동 모바일 UI 및 운동 모델 재설계

## 개요

이 문서는 `오늘운동` 서비스의 화면 구조와 운동 데이터 모델을 함께 재설계하기 위한 기준 문서다.

본 문서에서 제품명은 `오늘운동`으로 통일해 표기한다.

핵심 목표는 다음 네 가지다.

1. 운동 카테고리를 `웨이트 / 맨몸운동 / 유산소` 3개로 확장한다.
2. 화면 전반의 문장을 짧게 정리하고, 가능한 한 한 줄 안에서 읽히게 만든다.
3. 영어 중심 문구를 가능한 한 한글로 바꾸고, 상태 문구도 더 짧게 정리한다.
4. 아이폰 17 Pro Max 사파리 기준으로 스크롤을 최소화하는 모바일 구조로 화면을 다시 잡는다.

이번 작업은 단순 스타일 수정이 아니라, 데이터 모델과 UI 구조를 함께 손보는 전면 리팩터링이다.

## 제품 목표

사용자는 한 손으로 앱을 열고, 지금 해야 할 운동을 보고, 최소 입력만으로 운동을 기록하고, 최근 변화까지 빠르게 확인할 수 있어야 한다.

이를 위해 앱은 다음 원칙을 따른다.

- 화면마다 핵심 행동은 하나만 전면에 둔다.
- 카드 하나가 너무 많은 정보를 품지 않게 나눈다.
- 설명 문장은 최대한 짧게 쓰고, 길면 두 문장으로 나눈다.
- 모바일에서 자주 보는 정보는 접기 없이 먼저 보이게 하고, 덜 중요한 정보는 2차 영역으로 내린다.
- 카테고리 이름과 입력 방식이 섞이지 않도록 명확히 구분한다.

## 대상 화면과 제약

기준 기기는 아이폰 17 Pro Max의 사파리 브라우저다.

이번 리팩터링의 UI 제약은 다음과 같다.

- 기본 화면 폭에서 가로 스크롤은 허용하지 않는다.
- 세로 스크롤은 완전히 없애기보다, 첫 화면에서 가장 중요한 정보와 행동이 먼저 보이도록 줄인다.
- 첫 진입 시 각 페이지의 핵심 카드 1개와 보조 카드 1~2개 정도가 위쪽 화면 안에 들어오도록 설계한다.
- 카드 제목, 보조 설명, 버튼 문구가 동시에 여러 줄로 무너지지 않게 텍스트 길이와 폭을 제한한다.

## 현재 문제

현재 구조는 `웨이트 / 러닝` 이분법에 강하게 묶여 있다.

- 운동 유형, 루틴 유형, 기록 방식이 한 값으로 함께 묶여 있다.
- `플랭크` 같은 맨몸 동작도 내부적으로는 웨이트 취급이라 화면 의미가 어색하다.
- 유산소는 사실상 `러닝` 중심으로 표현되어 다른 유산소 운동을 담기 어렵다.
- 카드 내부 설명 문장이 길고, 페이지마다 정보 밀도가 일정하지 않다.
- 영어 라벨이 일부 남아 있고, 화면 문구 톤도 통일되지 않았다.
- 상단 헤더와 카드 조합이 커서 핵심 정보보다 장식이 먼저 보이는 페이지가 있다.

## 재설계 목표

### 1. 운동 구조 재정의

사용자에게 보이는 운동 카테고리는 아래 세 가지다.

- 웨이트
- 맨몸운동
- 유산소

하지만 내부 기록 엔진은 아래 두 가지 기록 방식으로 분리한다.

- 세트형
- 유산소형

이 분리를 통해 다음을 달성한다.

- 웨이트와 맨몸운동은 서로 다른 카테고리로 보인다.
- 두 카테고리는 모두 세트형 기록 구조를 공유한다.
- 웨이트만 중량을 가진다.
- 맨몸운동은 중량 없이 `세트 / 횟수 / 휴식`만 가진다.
- 유산소는 `종류 / 거리 / 시간 / 페이스`를 중심으로 기록한다.

### 2. UI 구조 단순화

모든 페이지는 공통적으로 아래 구조를 따른다.

1. 짧은 상단 요약
2. 첫 행동 카드
3. 보조 정보 카드
4. 세부 목록 또는 설정 카드

이 구조는 각 페이지가 처음 열렸을 때 사용자가 `지금 무엇을 해야 하는지`를 바로 이해하게 돕는다.

### 3. 텍스트 가독성 개선

문장 작성 원칙은 다음과 같다.

- 제목은 8~14자 안팎으로 유지한다.
- 설명 문구는 가능하면 한 줄, 길면 두 줄 안에 끝낸다.
- 복합 문장은 `중점 안내 + 보조 설명` 두 문장으로 나눈다.
- 영어 용어는 가능한 한 한글로 바꾼다.
- 버튼 문구는 동사 중심의 짧은 문구를 사용한다.

### 4. 스크롤 최소화

스크롤을 완전히 없애는 대신, 첫 화면 정보 밀도를 조정해 스크롤 필요량을 줄인다.

- 헤더 높이를 줄인다.
- 카드 안 요약 정보는 2열 또는 3열 요약칩으로 압축한다.
- 긴 리스트는 초기 표시 개수를 줄이고, 필요한 경우만 더 보게 한다.
- 반복적인 설명 문구는 줄이고, 상태나 수치 위주로 정리한다.

## 정보 구조

### 탭 구조

하단 탭 구조는 유지한다.

- 오늘
- 루틴
- 기록
- 설정

이 구조는 이미 사용 흐름과 잘 맞으므로 유지하고, 각 탭 내부의 정보 우선순위만 다시 정리한다.

### 페이지별 구조

#### 오늘

목표는 `오늘 무엇을 할지`와 `바로 시작 행동`을 가장 먼저 보여주는 것이다.

구성:

1. 오늘 요약 카드
2. 오늘 할 운동 카드
3. 짧은 추천 또는 상태 카드
4. 최근 기록 카드

상세 원칙:

- 주간 진행률, 연속 기록, 예정 운동 수를 한 카드 안에서 짧게 요약한다.
- 오늘 운동이 있으면 `운동 시작` 버튼을 가장 눈에 띄게 둔다.
- 추천 문구는 한두 문장으로 줄인다.
- 최근 기록은 2~3개만 먼저 보여준다.

#### 루틴

목표는 `루틴 탐색`보다 `루틴 편집과 분류`를 쉽게 만드는 것이다.

구성:

1. 루틴 요약 카드
2. 카테고리 필터 카드
3. 내 루틴 목록
4. 루틴 편집 시트

상세 원칙:

- 운동 추가 화면에서 `웨이트 / 맨몸운동 / 유산소`를 먼저 선택한다.
- 세트형과 유산소형 입력 폼을 한 화면에 섞지 않고, 선택한 항목 종류에 따라 입력 필드를 다르게 보여준다.
- 운동 검색, 장비 필터, 부위 필터는 한 화면에서 너무 길어지지 않도록 우선순위를 조정한다.
- 템플릿 영역은 기본 접힘 또는 짧은 요약 중심으로 정리한다.

#### 기록

목표는 `꾸준함 확인`과 `세부 기록 접근`을 빠르게 만드는 것이다.

구성:

1. 이번 주 요약 카드
2. 추이 카드
3. 건강 기록 카드
4. 지난 운동 목록

상세 원칙:

- 기간 선택, 지표 선택은 한글 라벨로 정리한다.
- 그래프 카드 제목과 설명은 짧게 유지한다.
- 지난 운동 목록은 날짜, 루틴명, 상태, 요약 수치만 먼저 보인다.
- 상세 화면에서 운동별 결과가 길어지면 항목 단위 카드로 나누되, 세트형과 유산소형 표시 규칙을 통일한다.

#### 설정

목표는 `보호 상태 확인`과 `백업/복원`을 짧은 흐름으로 만드는 것이다.

구성:

1. 저장 보호 상태
2. 백업 상태
3. 클라우드 백업
4. 앱 관리

상세 원칙:

- 경고 문구는 장문 설명 대신 핵심 위험과 권장 행동만 보여준다.
- `JSON`, `Supabase` 같은 용어는 필요한 곳만 남기고, 가능하면 한글 안내를 앞세운다.
- 위험 작업은 별도 영역으로 분리한다.

## 데이터 모델 설계

### 핵심 타입 방향

현재의 `strength / running` 중심 타입을 다음 방향으로 재설계한다.

#### Exercise

운동 메타 정보는 카테고리와 기록 방식을 분리한다.

- `category`: `weight | bodyweight | cardio`
- `recordMode`: `sets | cardio`
- `equipment`: 기존 장비 정보 유지
- `muscleGroup`: 세트형 운동 중심으로 유지

의도:

- 화면 분류는 `category`로 처리한다.
- 입력 UI와 기록 저장은 `recordMode`로 처리한다.

#### RoutineItem

루틴 항목은 공통 베이스와 두 가지 하위 구조로 나눈다.

- `SetBasedPlan`
  - 대상: 웨이트, 맨몸운동
  - 필드: `sets`, `targetReps`, `restSeconds`, `note`
  - 추가 필드: 웨이트일 때만 `targetWeightKg`

- `CardioPlan`
  - 대상: 유산소
  - 필드: `targetActivityLabel`, `targetDistanceKm`, `targetDurationMin`, `targetPaceMinPerKm`, `note`

여기서 `targetActivityLabel`은 `달리기`, `실내 자전거`, `빠른 걷기` 같은 실제 유산소 종류를 담는다.

#### WorkoutRecordItem

운동 기록도 같은 방식으로 둘로 나눈다.

- `SetBasedRecord`
  - 세트별 완료 여부
  - 실제 횟수
  - 실제 중량은 웨이트일 때만 저장

- `CardioRecord`
  - 실제 유산소 종류
  - 실제 거리
  - 실제 시간
  - 실제 페이스

### 루틴 유형 표시

루틴 전체의 단일 `kind` 값은 더 이상 중심 모델로 쓰지 않는다.

대신 루틴에 포함된 항목들을 보고 화면에서 계산해 보여준다.

예시:

- `웨이트 2개 포함`
- `맨몸운동 + 유산소`
- `유산소 중심`

저장 구조에서 기존 `kind` 필드는 점진적으로 축소하거나, 하위 호환을 위해 유지하더라도 화면 로직은 새 계산 함수를 사용한다.

### 하위 호환

기존 데이터는 다음 규칙으로 마이그레이션한다.

- 기존 `strength` 운동
  - 기본값은 `category = weight`, `recordMode = sets`
- 기존 `running` 운동
  - `category = cardio`, `recordMode = cardio`
  - 유산소 종류 기본값은 운동명 또는 기존 운동 이름을 사용
- 기존에 중량 0으로 쓰던 맨몸 동작
  - 사전 정의된 맨몸 동작 목록을 기준으로 `category = bodyweight`로 승격
  - 기록 방식은 그대로 `sets`

이 마이그레이션은 기존 사용자 데이터를 보존하는 것이 최우선이다.

## UI 컴포넌트 설계

### 공통 셸

`AppShell`은 다음 방향으로 단순화한다.

- 상단 헤더 높이 축소
- 배지, 제목, 설명의 폭 제한 강화
- 장식 요소 크기 축소
- 하단 탭 높이와 패딩을 모바일 사파리 안전 영역에 맞춰 재조정

### 카드 구조

카드는 역할에 따라 3종으로 정리한다.

- 요약 카드
- 행동 카드
- 목록 카드

각 카드의 공통 규칙:

- 제목은 짧게
- 보조 설명은 선택적
- 버튼은 카드마다 1차 행동 하나만 강조
- 칩, 상태 라벨, 수치 요약을 적극 사용

### 문장 줄바꿈 규칙

텍스트 렌더링 규칙:

- 제목과 큰 숫자 영역은 한 줄 우선
- 본문 설명은 `max-inline-size`를 더 보수적으로 잡는다
- 긴 설명은 데이터 가공 단계에서 두 문장으로 나눈다
- 목록 카드에서는 한 항목에 정보 줄을 2개 이상 넘기지 않도록 요약 문자열을 다시 설계한다

## 카피라이팅 기준

영어 문구는 화면에서 가능한 한 한글로 바꾼다.

예시:

- `Range` → `기간`
- `manual` → `직접 만든 루틴`
- `template` → `추천 루틴`
- `completed` → `완료`
- `partial` → `일부 기록`
- `skipped` → `건너뜀`

추가 원칙:

- `러닝`은 사용자에게 보이는 상위 카테고리로 쓰지 않는다.
- `유산소` 안에서 실제 운동명으로 `달리기`, `자전거`, `걷기` 등을 보여준다.
- 상태 문구는 `설명형`보다 `요약형`을 우선한다.

## 사용자 흐름

### 온보딩

- 운동 카테고리 선택: `웨이트 / 맨몸운동 / 유산소`
- 주당 운동 횟수
- 시작 방식 선택
- 추천 루틴 미리보기

추천 루틴은 선택한 카테고리에 맞게 구성한다.

예시:

- 웨이트 + 맨몸운동
- 맨몸운동 + 유산소
- 유산소 중심

### 루틴 작성

1. 루틴 이름 입력
2. 운동 카테고리 선택
3. 운동 추가
4. 운동별 입력 방식에 맞는 필드 편집

세트형 운동은 다음 순서로 보인다.

- 운동명
- 세트
- 횟수
- 휴식
- 중량(웨이트일 때만)

유산소는 다음 순서로 보인다.

- 유산소 종류
- 거리
- 시간
- 페이스

### 운동 기록

세트형과 유산소형은 입력 폼을 분리해 보여준다.

- 세트형: 세트 카드 반복
- 유산소형: 한 카드 안에서 종류, 거리, 시간, 페이스 입력

사용자는 입력 중에 불필요한 필드를 보지 않아야 한다.

## 에러 처리

- 마이그레이션 실패 시 기존 데이터는 절대 지우지 않는다.
- 새 필드가 없는 오래된 데이터는 안전한 기본값으로 읽는다.
- 유산소 기록은 `종류`와 `시간`을 필수로 하고, `거리`와 `페이스`는 선택으로 둔다.
- 유산소에서 거리와 시간이 모두 있으면 페이스는 자동 계산하되, 필요하면 사용자가 직접 수정할 수 있게 한다.
- 저장 실패 시 사용자에게 한글 메시지로 짧게 알린다.
- 삭제, 초기화, 복원은 위험 구역으로 분리하고 확인 절차를 유지한다.

## 테스트 전략

### 데이터 계층

- 기존 `strength / running` 데이터가 새 구조로 정상 해석되는지 검증
- 맨몸운동이 세트형으로 기록되고 중량 필드가 강제되지 않는지 검증
- 유산소 기록이 `종류 / 거리 / 시간 / 페이스` 구조로 저장되는지 검증
- 루틴 요약 라벨 계산이 정상 동작하는지 검증

### UI 계층

- 오늘, 루틴, 기록, 설정 첫 화면에서 핵심 카드가 상단에 바로 보이는지 검증
- 긴 문장이 여러 줄로 과하게 무너지지 않는지 검증
- 한글화 대상 문구가 영어로 남지 않는지 검증
- 세트형과 유산소형 입력 폼이 선택한 운동 카테고리에 맞게 바뀌는지 검증

### 회귀 위험

- 기존 세션 상세 화면
- 기존 추천 루틴 설치
- 백업/복원 데이터 형식
- IndexedDB 스키마 변경 후 기존 데이터 접근

## 성공 기준

- 사용자에게 보이는 운동 카테고리가 `웨이트 / 맨몸운동 / 유산소`로 일관된다.
- 맨몸운동은 중량 없이 세트형으로 기록된다.
- 유산소는 `종류 / 거리 / 시간 / 페이스` 중심으로 기록된다.
- 영어 문구가 눈에 띄게 줄고, 주요 화면 문구가 한글 중심으로 정리된다.
- 첫 화면에서 핵심 행동이 위쪽에 보이고, 스크롤 필요량이 줄어든다.
- 기존 데이터가 깨지지 않고, 기존 사용자도 앱을 이어서 사용할 수 있다.
